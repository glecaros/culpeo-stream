# HandleMediaAsync dispatch can never route frames in new sessions

**Severity:** High  
**Location:** `src/CulpeoStream.SourceGen/Generators/CodeGenerator.cs` lines 122–130; `src/CulpeoStream.Core/CulpeoSession.cs` line 454

## Description

The source generator emits a `HandleMediaAsync` switch-expression that dispatches incoming media frames by the stream ID strings declared in `[DeclareStream]` attributes — e.g. `"audio-in"`. However, the server's session core (`CreateNewSession`) ignores any client-supplied ID hint and unconditionally assigns cryptographically random opaque hex IDs to every stream:

```csharp
// CulpeoSession.cs line 454 — IdHint is never used here
var streams = init.Streams
    .Select(stream => new StreamState(GenerateOpaqueId(8), ...))
    .ToDictionary(stream => stream.Id, ...);
```

Generated adapter code:

```csharp
// Generated by CodeGenerator — stream IDs come from [DeclareStream]
=> streamId switch
{
    "audio-in" => OnAudioInAsync(payload, offset, cancellationToken),
    _ => OnUnknownStreamAsync(streamId, payload, offset, cancellationToken),
};
```

`ICulpeoStreamHandler.OnMediaFrameAsync` receives `CulpeoMediaFrameContext.StreamId` set to the server-assigned opaque hex value (e.g. `"a3f91c04"`). That value will never equal the `[DeclareStream]`-declared ID `"audio-in"`, so every incoming media frame falls into `OnUnknownStreamAsync`. All generated `On<Stream>Async` stubs are dead code for new sessions.

Session resumption partially mitigates this for resumed sessions because `FindExistingStream` will look up the stream by its opaque ID again, preserving the same ID — but users must already know that opaque ID from the previous session's `init-ack`, which the source generator provides no mechanism to capture or remember.

## Impact

Any application that implements a server-side `ICulpeoStreamHandler` annotated with `[CulpeoStreamHandler]` and relies on the generated `HandleMediaAsync` dispatch will silently receive all media in `OnUnknownStreamAsync`. There is no error, no diagnostic, no compile warning — the generated code compiles and runs, but the stream-routing feature is entirely non-functional for new sessions. This defeats the primary purpose of the source generator.

## Suggested Fix

Two approaches:

**Option A (preferred):** When building the generated `_culpeo_registeredStreams` static field, also embed the declared human-readable IDs. In `OnConnectedAsync`, the handler can use `session.Streams` (which carries the server-assigned IDs) to build a mapping from declared IDs to assigned IDs. Generate a helper method `MapServerStreamIds(IReadOnlyList<CulpeoStreamInfo> streams)` that performs this remapping once at session start, and have `HandleMediaAsync` dispatch against the mapped IDs rather than the static declared IDs.

**Option B (simpler):** Have the source generator emit a `RegisteredStreamIds` property that exposes the declared IDs so users can manually correlate them with `session.Streams` in `OnConnectedAsync`. Document that `HandleMediaAsync` must be called with the server-assigned ID and a pre-built map. Make the misleading implicit contract explicit.

Either way the current dispatch using raw `[DeclareStream]` IDs must not be promoted as a working routing mechanism until the ID mapping problem is solved.
